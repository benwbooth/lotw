# LotW Rust port — 6502 faithfulness audit

Automated audit: every routine in `src/game.rs` matched to its original 6502 code
in `rom/lotw.nes` and compared. Each finding independently re-verified against the
disassembly. Bug classes: dropped stores, dropped processor-flag (carry) results,
wrong constants, mistranslated/ inverted control flow, and spurious injected code.

## `audio_cmd_set_duty_instrument`
- **[high/dropped_flag]** X (the channel lane = sound_channel_offset, which can be 0x10/0x20/0x30/0x40) is dropped on the store: the Rust writes byte 6 of pulse-1's record (lane 0) regardless of which channel the command targets, corrupting pulse1's duty/volume shadow and failing to update the intended channel. Should be `set_sound_channel_byte(6, channel_offset, ...)`.
  - ROM: $FBCC LDA $99,X ; $FBCE AND #$3F ; $FBD0 ORA $00 ; $FBD2 STA $99,X — both the read and the write use the X-indexed lane ($99 = sound-channel base $93 + byte 6). The store is `STA $99,X`, i.e. write back to lane X, not lane 0.
  - Fix: At src/game.rs:11152, change the store target lane from `0` to `channel_offset`: `engine.state.set_sound_channel_byte(6, channel_offset, (((engine.state.sound_channel_byte(6, channel_offset) & crate::bits::LOW_6_BITS) | duty_bits) as u8 as i32));`
- **[high/dropped_flag]** Same dropped-lane bug as the byte-6 store: byte 7 must be written to the active channel lane X (channel_offset), but the Rust hardcodes lane 0, so the sustain/sweep value lands in pulse-1's record instead of the targeted channel. Should be `set_sound_channel_byte(7, channel_offset, ...)`.
  - ROM: $FBDC LDA $FDD2,Y ; $FBDF STA $9A,X — the sustain value is stored with `STA $9A,X` (lane X; $9A = base $93 + byte 7).
  - Fix: Change game.rs:11162-11168 to write lane `channel_offset` instead of `0`: `engine.state.set_sound_channel_byte(7, channel_offset, engine.state.byte(((SUSTAIN_TABLE + envelope_offset) as u16 as i32)));` (matching the 6502 `STA $9A,X`).
- **[medium/dropped_store]** The 6502 stores the merged duty byte and the cached sustain byte to the CURRENT channel's record ($99+X and $9A+X, X = lane offset 0/16/32/48/64). The Rust hardcodes the destination lane to 0 (set_sound_channel_byte(6,0,...) -> $99; set_sound_channel_byte(7,0,...) -> $9A), so for any channel other than pulse-1 the duty and sustain values are written into pulse-1's record instead of the issuing channel's. The envelope-offset store on line 11158-11160 correctly uses channel_offset, and audio_cmd_set_channel_flags correctly uses r.index, confirming `0` here is a translation error. The reads still use channel_offset, so the read/write targets are mismatched.
  - ROM: $FBCC LDA $99,X / $FBCE AND #$3F / $FBD0 ORA $00 / $FBD2 STA $99,X  (byte 6, indexed by X = sound_channel_offset $02); $FBDC LDA $FDD2,Y / $FBDF STA $9A,X (byte 7, indexed by X). X = sound_channel_offset (loaded at $FBB6 LDX $02 before JMP ($0006), not modified inside the handler).
  - Fix: In audio_cmd_set_duty_instrument (src/game.rs), change the two hardcoded lane-0 writes to use channel_offset: line 11150-11155 set_sound_channel_byte(6, channel_offset, ...) and line 11162-11168 set_sound_channel_byte(7, channel_offset, ...), matching $FBD2 STA $99,X and $FBDF STA $9A,X (X = sound_channel_offset).

## `check_final_exit_projectile_bounds`
- **[high/control_flow]** When scratch2 ($0A) >= 0xA1 the 6502 sets carry (SEC) to mark the projectile out of bounds. The Rust instead early-returns leaving r.carry untouched. Callers spawn_final_exit_projectile (line 939: `if r.carry == 0 { init }`) and update_final_exit_projectile_slot (line 965: `if r.carry != 0 { obj_state = 0 }`) treat carry=1 as out-of-bounds. Neither caller pre-clears/sets r.carry before the call (lines 936-937, 963-964), and the preceding project routines end in an ADC that can leave carry either way, so the high-Y out-of-bounds determination is dropped: a projectile with scratch2>=0xA1 will be (mis)treated as in-bounds and spawned/committed instead of rejected/killed. This is the SEC-misread-as-no-op class of bug.
  - ROM: $86B1: LDA $0A; $86B3: CMP #$A1; $86B5: BCS $86C1 ; and $86C1: SEC; $86C2: RTS. The scratch2(>=0xA1) path branches to SEC, explicitly SETTING carry (out-of-bounds), then RTS.
  - Fix: Set r.carry explicitly on every return path to match the 6502's SEC/CLC. Replace the three bare `return;` statements: scratch2>=161 -> `r.carry = 1; return;` (SEC @86C1); indirect_ptr_lo<241 -> `r.carry = 0; return;` (CLC @86C3); indirect_ptr_lo==0 -> `r.carry = 0; return;` (CLC @86C3); leave the trailing `r.carry = 1;` (SEC @86C1).

## `check_final_exit_trigger`
- **[high/control_flow]** The original game, once the final-exit (ending) trigger fires, bails out of the entire per-frame input/movement handler for that frame via the PLA;PLA non-local return. The Rust port drops that non-local return: after setting final_exit_flag it falls back into the dispatch loop and keeps processing Start, item use, input latching and movement that the original skips. This is the same dropped-non-local-return / shared-tail bug class as the known $D8E3 case.
  - ROM: $D64F (entry, JSR'd from $D436): after the six condition checks succeed, $D676 LDA #$01; $D678 STA $EC (final_exit_flag=1); then $D67A PLA; $D67B PLA; $D67C RTS. The double PLA discards both the JSR-$D64F return frame and the per-frame handler's own return frame, performing a non-local return TWO levels up — i.e. it aborts the rest of the per-frame handler ($D439 onward: Start-button test $D43B, JSR $D596 = tick_selected_item_effect at $D442, landing-timer/input at $D445+).
  - Fix: In check_final_exit_trigger, inside the success block (after `engine.state.final_exit_flag = 1;`) set `engine.lotw_nonlocal_handoff = 1;` (matching the PLA;PLA;RTS non-local return). Then in the state-1 dispatch arm immediately after `check_final_exit_trigger(engine, r);` at src/game.rs:230, add `if engine.lotw_nonlocal_handoff != 0 { return; }` so the rest of the per-frame handler is skipped, exactly as the original $D67A/$D67B PLA;PLA returns two levels up.

## `check_player_x_overlap`
- **[high/dropped_flag]** The 6502 routine always recomputes the carry on every return path; the Rust translation leaves r.carry unchanged on the same-tile and no-overlap paths. Because the sole structured caller `check_player_overlap` (matching $CE7C) reaches `check_player_x_overlap` only after `check_player_y_overlap` set r.carry=1, a no-overlap result from the X test (e.g. tile_delta in 2..254) leaves r.carry=1, so `check_player_overlap` line 4155 `if r.carry==0 { return; }` is not taken and it falsely sets overlap_flag=1 / carry=1 — a phantom collision the 6502 ($CE88 `BCC $CE8F`) would have rejected. The direct callers at game.rs:8223 and game.rs:8674 (6502 $ECD1, $EE5F, each `JSR $CE90` then `BCS`) likewise read the returned carry with no guaranteed carry-in, so the dropped CLC/SEC corrupts their branch decision.
  - ROM: $CE90 (matched: SEC; LDA $0F(indirect_ptr_hi); SBC $44(player_x_tile); writes carry on every exit). The no-overlap exits go to $CEB2 `CLC; RTS` (carry=0): reached by $CE9D `BCC $CEB2` (tile_delta in 2..254), $CEA4 `BEQ $CEB2` (==255, subtile==0), $CEA6 `BMI $CEB2` (==255, subtile negative). The same-tile exit $CE95 `BEQ $CEB4` goes to $CEB4 `SEC; RTS` (carry=1). The Rust drops both the CLC (carry=0) on the no-overlap paths and the SEC (carry=1) on the tile_delta==0 path, leaving r.carry at its stale entry value.
  - Fix: In check_player_x_overlap, set r.carry on every exit to match the 6502: replace the bare `return;` at line 4176 (tile_delta==0) with `{ r.carry = 1; return; }`; and set `r.carry = 0;` before the returns at lines 4190 (tile_delta 2..254), 4197 (delta==255, subtile==0) and 4200 (delta==255, subtile negative). Equivalently, add `r.carry = 0;` as the first statement of the function and then only the carry=1 paths (tile_delta==0 and the delta==255 overlap path) need an explicit set.

## `check_top_boundary_exit_clear`
- **[high/dropped_flag]** The two early-return branches each fall through to a shared carry-setting tail in the 6502 (this is the known mistranslation pattern). The airborne/jump path branches to $DCA6 which executes SEC (carry=1), and the indirect_ptr_lo!=0 path branches to $DCA4 which executes CLC (carry=0). The Rust replaces both with `return;`, dropping the carry store entirely, so callers see whatever stale carry was in r.carry on entry. The airborne/jump case is inverted vs intent: 6502 returns carry SET, and the Rust doc comment even wrongly states 'Returns carry clear (early) while airborne/jumping'. Fix: airborne/jump return should set r.carry = 1; indirect_ptr_lo!=0 return should set r.carry = 0.
  - ROM: 6502 routine at $DC87: $DC87 LDA $86; $DC89 ORA $4F; $DC8B BNE $DCA6  (airborne_flag OR jump_timer != 0). The branch target $DCA6 is `SEC; RTS` => carry SET. $DC8D LDA $0E; $DC8F BNE $DCA4 (indirect_ptr_lo != 0). The branch target $DCA4 is `CLC; RTS` => carry CLEAR. The fall-through tail at $DC99..$DCA2 reads the tile, AND #$3F, BEQ $DCA6 (SEC) else CLC, which the Rust does implement (r.carry = (tile==0)).
  - Fix: In check_top_boundary_exit_clear: change the airborne/jump early return (line 6162) to `r.carry = 1; return;` (matches $DCA6 SEC) and the indirect_ptr_lo!=0 early return (line 6166) to `r.carry = 0; return;` (matches $DCA4 CLC). Also fix the doc comment which wrongly says airborne/jumping returns carry clear.

## `dispatch_room_tile_action`
- **[high/control_flow]** Off-by-one in the equipped-item dispatch for tile id 62. The 6502 triggers the light spell on item 2, thrown weapon on item 3, magic blast on item 4; the Rust uses items 1/2/3. The lock path in the same routine uses `item == 7` (Rust line 14493) matching 6502 `DDEB CMP #$07` with no offset, proving item values are identical between Rust and ROM, so this is a true off-by-one, not a representation convention. Each of the three tile-item interactions would fire for the wrong equipped item.
  - ROM: $DD97 dispatch, tile==62 branch at $DE1A: `DE2B LDX $51,Y` loads idx, then `DE2D DEX; DE2E DEX; DE2F BEQ $DE3F` (idx==2 -> light-spell block at $DE3F), `DE31 DEX; DE32 BEQ $DE39` (idx==3 -> throw, JMP $DE9F), `DE34 DEX; DE35 BEQ $DE3C` (idx==4 -> magic blast, JMP $DEE8). The block bodies at $DE3F/$DE9F/$DEE8 match the Rust idx==1/2/3 bodies exactly (magic+alignment in-place conversion / build_direction_velocity offset 1 timer 16 / build_direction_velocity offset 8 timer 0 + magic).
  - Fix: In dispatch_room_tile_action's tile==62 branch, shift the three item comparisons up by one to match the ROM's DEX/DEX/BEQ sequence: change `if idx == 1` (line 14534) to `if idx == 2`, `if idx == 2` (line 14587) to `if idx == 3`, and `if idx == 3` (line 14628) to `if idx == 4`.
- **[high/control_flow]** The original item-7 path is: spend magic if available, otherwise fall back to spending a key to unlock the tile. The Rust drops the key-fallback entirely: with the magic-unlock item selected and zero magic, the tile can never be opened even though the player holds keys. This is a dropped/mistranslated branch (the BCC-on-success was rendered as an early return on failure, losing the fall-through to consume_key).
  - ROM: $DDE0 locked-tile path: $DDEB CMP #$07; $DDED BNE $DDF4; $DDEF JSR $E7F0 (consume_magic_point); $DDF2 BCC $DDF9. consume_magic_point ($E7F0) returns carry SET on failure, so when out of magic BCC is NOT taken and execution FALLS THROUGH to $DDF4 JSR $E86F (consume_key) / $DDF7 BCS $DE18. The original tries magic first and, if no magic, falls back to spending a key.
  - Fix: At src/game.rs:14493-14508, make the item==7 magic-failure case fall through to consume_key instead of returning. E.g.: `if item == 7 { r.index = engine.state.selected_item_slot as u8; consume_magic_point(engine, r); } if (item != 7 || r.carry != 0) { consume_key(engine, r); if r.carry != 0 { r.carry = 1; return; } }` — i.e. only return-as-solid when the key spend also fails (mirroring $DDF2 BCC success / fall-through to $DDF4 JSR $E86F / $DDF7 BCS $DE18).
- **[high/wrong_constant]** Every branch of the carried-item dispatch is shifted down by one: the original triggers the light spell on item 2, the thrown weapon on item 3, and the magic blast on item 4, while item 1 falls through to SEC/RTS (no action). The Rust triggers them on items 1, 2, 3. The inner-block logic (player_magic+alignment for light, speed 1/timer 16 for throw, speed 8/timer 0/consume-magic for blast) matches the $DE3F/$DE9F/$DEE8 blocks exactly, confirming the only error is the dispatch constant being off by one.
  - ROM: $DE1A interactable path: $DE2B LDX $51,Y (raw item id); $DE2D DEX; $DE2E DEX; $DE2F BEQ $DE3F (item==2 -> light-spell block that checks $59/player_magic & alignment); $DE31 DEX; $DE32 BEQ $DE39->JMP $DE9F (item==3 -> throw, speed LDY #$01, obj_timer #$10); $DE34 DEX; $DE35 BEQ $DE3C->JMP $DEE8 (item==4 -> blast, LDY #$08, obj_timer #$00, JSR $E7F0). The DEX;DEX;BEQ chain dispatches on item == 2, 3, 4, not 1, 2, 3. The same item_slot value is compared raw (CMP #$07) in the locked-tile path, so values are not pre-decremented.
  - Fix: In dispatch_room_tile_action change the id==62 carried-item dispatch keys to match the 6502 DEX chain: `if idx == 2` for the light spell (was 1), `if idx == 3` for the thrown weapon (was 2), and `if idx == 4` for the magic blast (was 3).
- **[medium/control_flow]** Dropped fall-through. In the original, equipping the magic-lockpick item (7) on a locked tile tries magic first and, when out of magic, falls through to also try spending a key. The Rust port short-circuits with `return` on magic failure, so a player with no magic but a key in hand can no longer open the lock.
  - ROM: $DDE7: `DDE9 LDA $51,X; DDEB CMP #$07; DDED BNE $DDF4` (item!=7 -> key path). For item==7: `DDEF JSR $E7F0` (consume_magic); `DDF2 BCC $DDF9` (carry clear/success -> spawn). On carry set (no magic) execution FALLS THROUGH to `DDF4 JSR $E86F` (consume_key); `DDF7 BCS $DE18` (no key -> SEC; RTS solid); otherwise spawns the removal projectile.
  - Fix: In dispatch_room_tile_action (lines 14493-14508 of src/game.rs), mirror the 6502 fall-through: on item==7 call consume_magic_point and if it fails (r.carry != 0) do NOT return; fall through to consume_key, returning solid only if the key also fails. Example: let need_key = if item == 7 { consume_magic_point(engine, r); r.carry != 0 } else { true }; if need_key { consume_key(engine, r); if r.carry != 0 { r.carry = 1; return; } } then continue to the projectile spawn.

## `main_loop_dispatch`
- **[high/control_flow]** The branch direction is inverted. The 6502 keeps running the scripted final-exit cutscene loop as long as the player is alive (BNE back to loop top while health!=0), exiting only when the player dies (health==0). The Rust breaks out when health!=0, i.e. it exits the loop on the very first iteration while the player is alive and never runs the cutscene body across frames. The condition should be `if engine.state.player_health == 0 { break; }` (loop while health != 0).
  - ROM: $C103: LDA $58 ($58=player_health); $C105: BNE $C0D4. $C0D4 is the top of the inner loop (JSR $CC43 read_buttons). The ROM loops back WHILE player_health != 0 and only falls through to exit when player_health == 0.
  - Fix: Change src/game.rs line 12693 from `if engine.state.player_health != 0 { break; }` to `if engine.state.player_health == 0 { break; }` (loop while health != 0, mirroring $C105 BNE $C0D4).

## `run_player_death_or_continue_flow`
- **[high/control_flow]** The branch direction is inverted. In the ROM, continue_timer bit7 CLEAR routes to the extra-life-token check, and bit7 SET routes to the increment-and-resume path. The Rust does the opposite: it runs the extra-life check when bit7 is SET and increments when bit7 is CLEAR. This swaps which armed/disarmed state grants a free resume vs. consumes an extra-life item, producing wrong death/continue behavior.
  - ROM: $935A: LDA $37 ; $935C: BPL $9363 (branch taken when bit7 CLEAR) -> $9363: LDX $55; LDA $51,X; CMP #$0C ... (the extra-life item check). The fall-through at $935E (taken when bit7 SET) is `INC $37` (continue_timer++) followed by `JMP $B372` (resume: animate_health_refill, pose 25, switch song, LDX #$00 / r.index=0).
  - Fix: At src/game.rs:13829 invert the condition to match the BPL (branch-on-bit7-clear): change `if (engine.state.continue_timer & ((crate::bits::BIT7) as u8)) != 0 {` to `... == 0 {`. This routes the bit7-CLEAR case to the extra-life item-consume/resume check and the bit7-SET case to the `continue_timer += 1` (then resume) branch, matching ROM $935C BPL $9363.

## `tick_inactive_actor_slot`
- **[high/control_flow]** The 6502 inactive-slot tick only attempts a spawn (consuming RNG via $CC64 for the random Y/X position, running the overlap/terrain probes, and computing the member mask) once the per-slot countdown timer drops below 60. The Rust port dropped the `CPX #$3C; BCS RTS` guard, so it runs the full spawn-position probe and consumes RNG every frame the slot is empty. This both desyncs the RNG stream and changes actor spawn timing/placement.
  - ROM: $E9A5: DEC $F3 ; $E9A7: LDX $F3 ; $E9A9: CPX #$3C ; $E9AB: BCS $E9E6  (where $E9E6: RTS). The 6502 returns early whenever the decremented spawn timer obj_timer ($F3) is >= 60 (0x3C), only running the spawn/RNG logic during the final 60 frames of the countdown.
  - Fix: After the decrement at line 7449, add the early-return guard: `if engine.state.obj_timer >= 60 { return; }` (matching CPX #$3C / BCS $E9E6), before reading actor_timer and running the spawn body.

## `tick_scripted_player_motion`
- **[high/dropped_store]** When the diagonal move is blocked, the routine clears the horizontal delta and retries a vertical-only (straight-down) move. In the 6502 original, if that retry SUCCEEDS (carry clear, BCC $8CA1) it falls into the shared commit tail at $8CA1, which stores the new position into player_x_fine ($43) and player_y ($45), re-evaluates fall state, and finishes the frame. The Rust mistranslated this success branch as a bare `return;` (line 1390), DROPPING the position commit, fall-state update, and frame-finish. This is the exact known bug class (branch-to-shared-tail-store turned into a bare return). The successful vertical drop is silently discarded, so the scripted player will not actually move down even though the original commits the move. The first try_move just above (line 1380-1384) correctly calls commit on success; only this second retry is wrong.
  - ROM: Bank 13: $8C61: LDA #$00; STA $49; $8C65: JSR $ADC7 (try_move); $8C68: BCC $8CA1; $8C6A: JMP $ACAF. The BCC target $8CA1 is commit_scripted_player_position ($8CA1: LDA $0E; STA $43 (player_x_fine); LDA $0A; STA $45 (player_y); JSR $ADE4 (update fall state); JMP $ACBB (finish frame)). $ACAF==$8CAF is cancel. JSR $ADC7 ($8DC7) returns carry clear = move OK (matches r.carry==0).
  - Fix: In src/game.rs line 1389-1391, replace the bare `if ((r.carry) == 0) { return; }` with `if ((r.carry) == 0) { commit_scripted_player_position(engine, r); return; }` so the successful vertical-only drop commits the new position (matching 6502 BCC $8CA1).

## `update_object_terrain_probe`
- **[high/dropped_store]** This is the known dropped-shared-tail bug class. The Rust translated branches to the $F1D3 blocked tail as plain `return;`, dropping both the obj_move_scratch (0xF0) reset to 0 and the conditional obj_cooldown (0xF1)=move_scratch-4 store on every non-falling exit (including the cooldown-skip path). The fall counter is therefore never cleared and the jump cooldown never armed when the object lands/is blocked, diverging from the original each frame the probe bails.
  - ROM: $F179 (update_object_terrain_probe): the cooldown skip `F179: LDA $F1; F17B: BNE $F1D3` and every blocked/overlap/empty branch (`F1A5: BCS $F1D3`, `F1B4: BEQ $F1D3`, `F1BB: BEQ $F1D3`, `F1C2: BCS $F1D3`, `F1CD: BCS $F1D3`) branch to the SHARED TAIL at $F1D3, which is not a bare return: `F1D3: LDA $F0; F1D5: CMP #$0C; F1D7: BCC $F1DE; F1D9: SEC; F1DA: SBC #$04; F1DC: STA $F1; F1DE: LDA #$00; F1E0: STA $F0; F1E2: SEC; RTS` (i.e. if obj_move_scratch>=12 set obj_cooldown=obj_move_scratch-4, then always obj_move_scratch=0, carry=1). The success path $F1CF: INC $F0; CLC; RTS is the only one the Rust models (mark_probe_clear).
  - Fix: Replace the bare `return;` bail paths (lines 9192, 9203, 9211, 9226, 9229, 9236, 9246) with a shared tail helper that mirrors $F1D3: `fn probe_blocked_tail(engine, r) { if engine.state.obj_move_scratch >= 0x0C { engine.state.obj_cooldown = engine.state.obj_move_scratch - 4; } engine.state.obj_move_scratch = 0; r.carry = 1; }` and call it before returning on each bail path (including the cooldown-skip at line 9191-9193).
- **[high/dropped_store]** The 6502 'blocked / resting on floor' tail ($F1D3) commits two stores (conditionally obj_cooldown=$F0-4 when $F0>=0x0C, and always obj_move_scratch=0) and returns carry set. The Rust replaces every branch to $F1D3 with a bare `return;`, dropping both stores and the carry result, so a large object that lands or whose probe is blocked never has its cooldown latched or its fall counter cleared. Additionally the active-actor floor bail ($F19B `BCS $F1CF`, when obj_y_pixel>=0xB0) and the tile-aligned-clear bail ($F1C6 `BEQ $F1CF`) target $F1CF (`INC $F0; CLC` = mark_probe_clear) but the Rust returns bare at lines 9203 and 9241 instead of incrementing obj_move_scratch / clearing carry.
  - ROM: 6502 routine at $F179. $F17B `BNE $F1D3`, and $F1A5/$F1B4/$F1BB/$F1C2/$F1CD all branch to the tail at $F1D3, which is NOT a plain return: `F1D3: LDA $F0; CMP #$0C; BCC $F1DE; SEC; SBC #$04; STA $F1` (if obj_move_scratch>=0x0C set obj_cooldown=obj_move_scratch-4) `F1DE: LDA #$00; STA $F0` (obj_move_scratch=0) `F1E2: SEC; RTS` (carry set). This entire 'land/rest' tail is dropped in the Rust.
  - Fix: Add a shared rest tail helper matching $F1D3 and call it (instead of bare return) on the six blocked/cooldown/overlap/solid paths, and call mark_probe_clear() (instead of bare return) on the two $F1CF paths. Concretely: replace the bare returns at game.rs:9192, 9211, 9226, 9229, 9236, 9246 with a call to a new fn object_probe_rest(engine,r){ if engine.state.obj_move_scratch>=0x0C { engine.state.obj_cooldown = engine.state.obj_move_scratch - 4; } engine.state.obj_move_scratch = 0; r.carry = 1; } then return; and replace the bare returns at 9203 and 9240 with `mark_probe_clear(engine, r); return;`.

## `update_room_actors`
- **[high/control_flow]** On the defeated/dying large-actor (boss) path the original first calls $F430 — the boss death-animation routine that increments obj_state ($EE), spawns explosion sprites (JSR $C540/$C135), and on completion zeroes obj_state, sets obj_timer ($F3)=$F0 and pending_special_exit ($EB)=1 to despawn the boss and flag the room exit — before updating facing ($F53B) and body tiles ($F552). The Rust port omits the $F430 call entirely, so a defeated boss never runs its death progression (no explosion, never despawns, never sets the room-exit flag).

  - Fix: In the state-1 obj_state-bit7-set branch (src/game.rs:7348-7351), add a call to the $F430 boss death-progression helper before the facing/body-tile calls, e.g. `progress_large_actor_death(engine, r); update_large_actor_facing_from_velocity(engine, r); animate_large_actor_body_tiles(engine, r);` — and port $F430 (explosion spawn, obj_state increment, and the completion path that sets obj_state=0, obj_timer=240, pending_special_exit=1) if no equivalent helper exists yet.
